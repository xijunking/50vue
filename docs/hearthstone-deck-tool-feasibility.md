# 炉石卡组检索工具需求可行性方案

## 1. 结论

基于当前项目技术栈与目录结构，这个需求在本项目中实现的可行性较高，适合以“新增一个独立工具页”的方式落地。

原因如下：

- 当前项目已经是 `Vue 3 + TypeScript + Vite + Element Plus`
- 项目里已经存在列表展示、搜索输入、路由拆分、独立页面等实现基础
- 需求核心是前端本地读取结构化数据并做展示、过滤、复制，不依赖后端
- 当前版本的数据来源明确为外部整理后的 JSON，能明显降低复杂度

整体判断：

- 技术难度：低到中
- 改动范围：可控
- 首版实现成本：较低
- 与现有项目兼容性：高

---

## 2. 结合当前项目的推荐落地方式

### 方案定位

建议将该功能作为一个新的独立页面接入当前项目，而不是嵌入现有首页某个复杂模块中。

推荐原因：

- 当前项目本身就是多页面/多功能集合型结构，新增一个工具页最自然
- 路由模块已经按功能拆分，适合继续扩展
- 该需求本质是“工具页”，和现有的“网址导航”“小游戏”“项目页”定位一致
- 独立页面可以减少对已有页面的干扰，后续也更容易继续扩展

### 推荐接入形式

建议新增：

- 一个卡组工具页面，路由固定使用 `/lscs`
- 一个对应的数据文件，如 `src/data/hearthstone-decks.json`
- 如有必要，再补一个类型定义文件，例如 `src/types/deck.ts`

如果后续希望从首页进入，可以在首页新增一个入口卡片；如果当前只是内部使用，也可以先只加路由，不着急放首页入口。

---

## 3. 当前版本最可行的实现方案

结合本次补充，建议将实现过程明确拆成两步，而不是直接从页面入手。

### 3.1 第一步：先实现 txt 转 json

第一步先解决数据整理问题，把人工收集的 txt 内容稳定转换成结构化 JSON。

这一阶段的目标：

- 输入人工整理后的 txt 文本
- 按约定格式解析卡组信息
- 输出页面可直接消费的 JSON 文件

这样拆分的好处：

- 先把数据源稳定下来
- 降低页面层处理脏数据的复杂度
- 让“数据整理”和“页面展示”职责分离

当前版本建议的链路应当是：

`txt 原始整理 -> 转换脚本 -> json 数据文件 -> 页面读取展示`

这一阶段的重点不是做“自由文本智能识别”，而是做“约定格式下的稳定转换”。

### 3.2 数据层方案

当前版本最稳妥的做法是“先用 txt 生成静态 JSON，再由页面读取 JSON”。

推荐数据结构如下：

```json
[
  {
    "id": "dk-plague-001",
    "name": "瘟疫DK",
    "className": "死亡骑士",
    "author": "某某up主",
    "sourceNote": "B站简介整理",
    "deckCode": "AAECAfHhBA..."
  }
]
```

字段建议：

- `id`：唯一标识，便于列表渲染和后续扩展
- `name`：卡组名称
- `className`：职业
- `author`：来源作者或 up 主
- `sourceNote`：补充来源说明，可选
- `deckCode`：卡组代码

这样设计的优点：

- 足够简单，完全覆盖首版需求
- 便于人工维护
- 后续扩展版本、强度、备注、发布日期等字段时兼容性好

### 3.3 txt 转 json 的建议方式

为了让第一步更容易落地，建议先约定 txt 的输入格式。

推荐每条卡组至少包含：

- 卡组名称
- 职业
- 来源作者
- 卡组代码
- 可选备注

推荐原则：

- 格式尽量固定
- 一条卡组一段
- 字段名统一

这样做的原因：

- 方便人工从 B 站评论区或简介中整理
- 转换脚本只需处理一种稳定结构
- 后续页面数据质量更容易保证

### 3.4 页面层方案

页面建议保持单页工具结构，避免首版做复杂信息架构。

推荐布局：

1. 页面标题区
2. 搜索框
3. 卡组列表区
4. 空结果提示

每条卡组卡片展示：

- 卡组名称
- 职业标签
- 来源作者/备注
- 卡组代码
- 复制按钮

页面设计要突出一个非常明确的核心：

- 简洁
- 方便
- 一键复制

推荐理由：

- 与需求完全一致
- 操作路径短
- 列表+搜索的交互成本最低
- 适合当前“个人效率工具”的定位
- 能直接服务“搜索 -> 点击 -> 复制”的最短流程

因此当前版本不建议把页面做成内容型页面，而应做成工具型页面：

- 视觉尽量干净
- 交互尽量直接
- 复制动作尽量靠前

### 3.5 搜索方案

当前版本建议使用前端本地实时过滤，不需要单独搜索服务。

搜索逻辑建议：

- 输入框内容变化后，直接在内存中的 JSON 列表上过滤
- 支持对以下字段做模糊匹配：
  - 卡组名称
  - 职业
  - 作者
  - 来源备注

推荐实现思路：

- 将用户输入统一做 `trim + toLowerCase`
- 将每条卡组的可搜索字段拼接或逐项比对
- 返回匹配结果

搜索体验目标：

- 尽量少输入就能找到目标卡组
- 尽量减少滚动查找成本

这个方案适合当前阶段的原因：

- 数据量预计不大
- 实现简单
- 响应速度足够快
- 后续即使需要增加筛选条件，也容易升级

### 3.6 复制方案

推荐直接使用浏览器 Clipboard API 实现一键复制。

交互建议：

- 点击“复制卡组”
- 成功后显示轻提示，例如“已复制”
- 失败时显示“复制失败，请手动复制”

这部分应当作为页面的核心能力重点体现，建议：

- 复制按钮位置固定且醒目
- 用户无需进入详情页即可完成复制
- 卡组代码和复制按钮保持就近布局

这部分在当前项目中也比较顺手，因为已经使用了 Element Plus，可以直接沿用现有消息提示风格。

## 4. 与当前项目结构的匹配点

从现有代码看，这个需求和项目结构的匹配度较高，主要体现在：

### 4.1 路由扩展方便

当前项目已经使用模块化路由：

- `src/router/modules/base.ts`
- `src/router/modules/projects.ts`
- `src/router/modules/games.ts`

因此后续可以自然增加一个新的功能路由模块，或直接加到 `base.ts` 中。

### 4.2 已有“搜索 + 列表页”经验

现有 `NavPage.vue` 已经具备以下模式：

- 搜索输入
- 列表过滤
- 页面级独立布局

这说明卡组工具页在交互形态上与当前项目经验一致，很多组织方式可以直接借鉴。

### 4.3 已有 UI 组件基础

项目已集成 `Element Plus`，因此以下能力都不需要额外引入库：

- 输入框
- 按钮
- 标签
- 消息提示
- 空状态

这使得首版可以快速完成，不需要额外评估组件库成本。

## 5. 首版建议范围

建议首版严格控制在以下范围内：

### 必做

- txt 转 json 的本地转换能力
- 静态 JSON 数据读取
- 卡组列表展示
- 关键词搜索
- 一键复制卡组代码
- 基础空状态提示

### 建议但可选

- 职业标签颜色区分
- 复制成功状态反馈
- 卡组代码文本截断与展开展示

### 当前不建议加入

- 后台录入
- 自动抓取 B 站评论/简介
- 多维度高级筛选
- 收藏/本地持久化
- 用户系统

原因很明确：这些内容会显著拉高复杂度，并且不属于当前需求的核心价值。

## 6. 推荐目录组织

如果后续开始实现，推荐的文件组织可以是：

```text
src/
  data/
    hearthstone-decks.json
  types/
    deck.ts
  views/
    tools/
      HearthstoneDeckTool.vue
  router/
    modules/
      tools.ts
scripts/
  deck-txt-to-json.ts
raw-data/
  hearthstone-decks.txt
```

说明：

- `data`：存放人工整理后的结构化卡组数据
- `types`：约束数据结构，降低后续维护成本
- `views/tools`：统一放工具页，便于后面继续扩展别的效率工具
- `router/modules/tools.ts`：把新工具类页面从基础路由里分出来，结构更清晰
- `scripts`：存放 txt 转 json 的本地转换脚本
- `raw-data`：存放人工整理的原始 txt 文本

如果你希望尽量少改目录，也可以先不建 `types` 和 `tools.ts`，首版完全可以简化。

## 7. 可行性评估与风险

### 可行性高的点

- 无需后端
- 无需数据库
- 无需登录权限
- 无需复杂状态管理
- 与现有 Vue 项目完全兼容

### 主要风险点

#### 1. 数据维护依赖人工整理

当前版本数据来源是人工从评论区或简介整理，因此最大的不确定性不在前端实现，而在数据清洗质量。

可能问题：

- 卡组名称不统一
- 作者名称格式不统一
- 卡组代码复制时夹带无关字符

建议：

- 在 JSON 入库前做一次人工校验
- 统一字段命名与内容格式

#### 2. 搜索体验依赖字段规范

如果同一种职业出现多个写法，比如“法师”“Mage”，搜索体验会受影响。

建议：

- 首版统一使用中文职业名
- 如后续有需要，再增加别名字段

#### 3. 数据量继续增长时可读性下降

如果未来卡组数量明显增多，只有搜索可能不够。

不过这是后续阶段问题，当前版本先做搜索已经足够合理。后续再逐步增加职业筛选、版本筛选即可。

---

## 8. 最推荐的首版方案结论

最推荐的方案可以概括为一句话：

在当前 Vue 项目中，先实现 txt 转 json 的数据整理链路，再新增一个路由为 `/lscs` 的炉石卡组工具页，通过本地 JSON 读取卡组数据，使用前端实时搜索过滤列表，并通过醒目的复制按钮完成卡组代码的一键复制。

这个方案的优势是：

- 实现快
- 风险低
- 与现有项目兼容度高
- 完整覆盖当前需求
- 为后续扩展预留空间

---

## 9. 后续实现建议顺序

如果下一步要进入开发，建议按下面顺序推进：

1. 先约定 txt 输入格式
2. 明确 JSON 数据结构
3. 准备一小批真实卡组样例 txt
4. 完成 txt 转 json
5. 完成 `/lscs` 页面静态展示
6. 接入搜索
7. 接入复制功能
8. 最后再决定是否挂首页入口

这个顺序可以确保先把核心闭环跑通，再考虑入口和样式优化。

---

## 10. 本次结论

这个需求非常适合在当前项目中实现，而且建议优先采用“先 txt 转 json，再做 `/lscs` 独立工具页 + 本地搜索 + 一键复制”的轻量方案。

这会是当前版本投入产出比最高、也最符合产品目标的一条路线。
